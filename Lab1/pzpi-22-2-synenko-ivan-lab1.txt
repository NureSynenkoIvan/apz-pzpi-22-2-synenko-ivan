1. Бізнес-вимоги
   1. Передумови
Зараз Росія веде війну проти України, запускаючи хвилі ракет, керованих авіабомб і безпілотних літальних апаратів на об’єкти в Україні, здійснюючи терористичні атаки та загрожуючи важливим об'єктам.
В наш час безпілотна авіація набуває дедалі більшого значення у війні, слугуючи засобом для нанесення точних ударів за допомогою зброї, що порівняно з ракетами є дуже дешевою. Дрони доступні для відкритої покупки, тож у майбутньому ми, ймовірно, побачимо замахи та терористичні атаки з використанням безпілотників. У цьому світлі системи контрзаходів проти повітряних, особливо безпілотних, атак стають надзвичайно важливими для безпеки великих об’єктів.
   2. Бізнес-можливість
Багато державних і великих приватних об’єктів, таких як фабрики, електростанції, опалювальні станції, склади, є вразливими до атак дронів. Впровадження систем швидкого реагування є критично важливим для їхньої роботи в умовах війни.
У майбутньому об’єкти за межами України також можуть потребувати систем захисту від дронів через ризик терористичних атак, що може відкрити для нас новий великий ринок для поширення системи.
Наше дослідження показало, що на даний момент таких систем реагування ще не існує. Український уряд впровадив систему надсилання аварійних повідомлень, а кілька Telegram-каналів, таких як “TLK news”, “Куди летить” і “Повітряний простір України”, працюють як системи моніторингу в реальному часі, які повідомляють про загрози в повітрі. Однак це не системи екстреного реагування, а лише системи попередження для населення. Наскільки мені відомо, протиповітряне реагування не автоматизоване, та виконується силами протиповітряної оборони у тому числі за допомогою системи ситуаційної обізнаності “Дельта”. 
“Дельта” - це система сил безпеки і оборони України, яка інтегрує дані про супротивника на мапі, при чому дані беруться з різних джерел: від постачальників супутникових знімків, з радарів, сенсорів, GPS-трекерів, радіоперехоплення. Проте “Дельта” сконцентрована на більш загальних потребах військових, тоді як розроблюваний програмний продукт концентруватиметься саме на автоматичній реакції на загрозу. До того ж, “Дельта” є національною власністю України, що виключає її комерційне застосування.
2. Концепція рішення
   1. Окреслення концепції
Запропоноване рішення передбачає створення системи захисту, здатної захищати великі об'єкти від повітряних загроз, зокрема дронів. Ця система забезпечить автоматизацію виявлення загроз і безпрецедентно швидке реагування на них, створюючи безпечніше середовище для об'єктів та персоналу шляхом скорочення часу між виявленням загрози та реагуванням на неї. З запровадженням цієї системи на критично важливих об'єктах, таких як фабрики, електростанції та аеропорти, їх персонал отримуватиме негайні й надійні сповіщення про загрози, що дозволить завчасно вживати заходів для протидії можливим атакам.
   2. Головна функціональність
1. Автоматизоване виявлення дронів: Система автоматично виявлятиме активність дронів за допомогою даних, отриманих з радіоприймачів, визначаючи потенційні загрози автоматично.
2. Сповіщення та повідомлення в режимі реального часу: Персонал отримуватиме миттєві сповіщення та детальну інформацію про загрози на мобільні пристрої з Android (звичайний персонал) чи на рації (служба безпеки), що дозволить швидко та обґрунтовано реагувати.
3. Повідомлення та доступ на основі ролей: Система підтримуватиме сповіщення та доступ на основі ролей, розрізняючи простих працівників установи, диспетчерів, співробітників служби безпеки та спеціалізовані команди реагування (наприклад, команди з протидроновими рушницями).
4. Масштабованість для охоплення великих територій: Система має бути здатна захищати території площею понад 50 квадратних кілометрів, що дозволить забезпечити безпеку великих об'єктів, таких як аеропорти або промислові комплекси.
5. Журналювання даних для аналізу: Усі виявлені сигнали та реакції системи зберігатимуться для полегшення аналізу, що призвано підвищить точність і адаптивність системи з часом.
3. Рамки та обмеження
   1. Рамки первинного випуску
Первинним випуском передбачено створення серверу, IoT-клієнту, веб-частини та мобільної частини. 
Через закритість інформації про військові радари і їх АРІ у якості сенсорів були обрані радіоприймачі.
Серверна частина: збір і моніторинг інформації від радіоприймачів. Підіймання тривоги при появі підозрілих об’єктів. Збереження інформації про персонал установи, про існуючі радари і пристрої. Журналювання даних з радіоприймачів для подальшого аналізу. Можливість розширення для додавання модулів роботи з іншими типами сенсорів (радари, аудіосенсори, відеокамери, тощо).
IoT-клієнт: отримання інформації з радіоприймачів і надсилання її на сервер. Приймання повідомлення з серверу і програвання його. Імітація (mockup) роботи радіостанцій для отримання тестових даних і тестування реакції системи. 
Веб-частина: веб-застосунок, яким будуть користуватися диспетчери і адміністратор системи. Перегляд і редагування інформації про персонал. Демонстрація роботи радарів, можливість переглядати журнали з попередньо записаною інформацією. Можливість ручного об’явлення тривоги (на випадок хімічної атаки чи інших сценаріїв загроз). Можливість показу QR-коду, який потрібен для логіну (для додаткових мір безпеки). Надсилання пуш-повідомлень на мобільні застосунки. Серверний запит до мобільних застосунків, який поверне їх місцезнаходження, що дозволить розуміти оперативну обстановку.  Редагування тексту пуш-повідомлень.
Мобільний застосунок: можливість зареєструватися, можливість сканування QR-коду з ціллю логіну (для додаткових мір безпеки). Отримання пуш-повідомлень, надсилання актуальної інформації про місцезнаходження на запит від серверу. 


   2. Рамки наступних випусків 
Наступні випуски передбачають розробку веб-частини, мобільного клієнту, а також розширення серверного функціоналу. 
Серверна частина має мати можливість розширення для додавання модулів роботи з іншими типами сенсорів (радари, аудіосенсори, відеокамери, тощо), а також роботи з різними типами систем сповіщень (наприклад, зовнішніми сиренами).
Збір і журналювання інформації з сенсорів має дозволити у майбутньому навчити модель машинного навчання на зібраній інформації і додати модуль, який буде аналізувати інформацію з сенсорів за допомогою моделі машинного навчання і видавати сповіщення про можливу небезпеку диспетчерам, посилаючи їх у систему через messaging system Spring Integration. Це може дозволити виявляти небезпеку по непрямим факторам, які не завжди помітні людям.


   3. Обмеження і виключення 
Перший випуск навмисно не включає роботу з радарами через закриту природу інформації про військові та цивільні потужні радари, а також дороговизну радарного обладнання. Замість цього у якості сенсорів будуть виступати машини з модулем SDR (Software-Defined Radio).  
Модульна природа системи має дозволити у майбутньому додати модуль роботи з радарами, як і з іншими сенсорами, якщо буде відомо, з якими саме радарами вона має працювати.


4. Бізнес-контекст
4.3 Технології, які будуть використані у розробці
* Сервер: Java для серверної частини. Spring framework для организації коду, Spring Web для REST API, Spring Integration для уніфікації зовнішніх повідомлень про загрозу. Docker для контейнеризації і полегшення розгортання. 
* База даних: MongoDB для зберігання даних, Spring Data JPA для взаємодії з базою.
* IoT-технології: модулі для приймання і обробки інформації з радіоприймачів. Raspberry pi з модулем SDR. 
* Фронтенд: TypeScript, фреймворк React, HTML+CSS
* Мобільний застосунок: Kotlin
